# Roadmap редактора QA-чек-листов

Документ описывает развитие редактора. Основные функции этого roadmap реализованы
в текущей версии; пункты, зависящие от конкретной Jira, требуют проверки на реальном
корпоративном экземпляре.

## 1. Отмена и повтор действий

### Пользовательский сценарий

В шапке приложения появятся кнопки:

- «Отменить»;
- «Повторить».

Пользователь сможет отменить или вернуть:

- изменение текста в ячейке;
- изменение статуса;
- добавление, удаление и перемещение строки;
- разделение таблицы на разделы;
- добавление, удаление и переименование колонки;
- добавление, удаление и перемещение раздела;
- импорт Jira-разметки;
- изменение метаданных отчёта.

### Реализация

- хранить историю снимков структурированного документа;
- объединять последовательный ввод текста в одну операцию, чтобы каждое нажатие клавиши
  не становилось отдельным шагом;
- ограничить историю, например, последними 100 операциями;
- поддержать сочетания `Ctrl/Cmd + Z` и `Ctrl/Cmd + Shift + Z`;
- не включать в историю переключение темы, открытие окон и другие действия интерфейса;
- сохранять текущий документ после undo/redo в локальный черновик.

## 2. Исправление предпросмотра в тёмной теме

### Проблема

В тёмной теме заголовки таблиц и отдельные элементы визуального предпросмотра могут
оставаться светлыми.

### Реализация

- убрать жёстко заданные светлые цвета из `.jira-preview`;
- добавить отдельные CSS-переменные предпросмотра:
  `--preview-surface`, `--preview-header`, `--preview-border`, `--preview-text`;
- применить их к таблицам, заголовкам, ячейкам и пустым значениям;
- проверить визуальное отображение всех пяти статусов;
- отдельно проверить длинные таблицы с горизонтальной прокруткой.

## 3. Полноэкранный режим чек-листа

### Пользовательский сценарий

Кнопка «Во весь экран» переводит приложение в рабочий режим, где остаются:

- компактный хедер;
- разделы и таблицы чек-листа;
- основные действия с таблицами;
- кнопка выхода из полноэкранного режима.

Скрываются:

- ссылка на задачу;
- окружение и общий итог;
- вводный текст;
- боковая сводка;
- необязательные элементы интерфейса.

### Реализация

- добавить отдельное состояние `focusMode`;
- применить CSS-класс к корневому контейнеру;
- расширить таблицы на всю доступную ширину;
- сохранить вертикальную и горизонтальную прокрутку;
- добавить выход по кнопке и клавише `Escape`;
- запоминать режим только в текущей вкладке, но не делать его постоянной настройкой;
- не использовать обязательный Browser Fullscreen API: режим должен работать даже
  без системного разрешения браузера.

## 4. Фрагменты кода внутри ячеек

### Пользовательский сценарий

В ячейку можно вставить блок кода, выбрать язык и получить:

- моноширинный шрифт;
- сохранение пробелов и отступов;
- перенос или горизонтальную прокрутку;
- подсветку синтаксиса;
- корректный экспорт в Jira.

### Модель данных

Содержимое ячейки следует перевести от простой HTML-строки к массиву блоков:

```text
Cell
 └─ blocks[]
     ├─ paragraph
     ├─ list
     └─ codeBlock
         ├─ language
         └─ code
```

### Реализация

- добавить команду «Вставить код» в панель форматирования ячейки;
- использовать Highlight.js или Shiki для визуальной подсветки;
- хранить исходный код отдельно от подсвеченного HTML;
- для Jira Data Center экспортировать поддерживаемый `{code:language=...}`-макрос;
- для Jira Cloud создавать ADF-узел `codeBlock` с атрибутом языка;
- при импорте распознавать Jira code-макросы и ADF code blocks;
- экранировать символы, которые конфликтуют с Wiki-разметкой таблиц.

## 5. Изображения и вложения Jira

### Пользовательский сценарий

Изображение можно:

- вставить из буфера обмена;
- перетащить в ячейку;
- выбрать с диска;
- переименовать;
- удалить или заменить;
- открыть в полном размере.

При публикации приложение:

1. загружает изображения во вложения задачи;
2. получает подтверждённые Jira-имена или идентификаторы;
3. заменяет локальные элементы ссылками на вложения;
4. создаёт комментарий только после успешной загрузки;
5. показывает результат для каждого файла.

### Модель данных

```text
Attachment
 ├─ id
 ├─ originalName
 ├─ jiraFileName
 ├─ mimeType
 ├─ size
 ├─ localBlob
 ├─ uploadStatus
 └─ jiraAttachmentId
```

### Хранение

- метаданные документа остаются в `localStorage`;
- бинарные файлы хранятся в IndexedDB;
- токен Jira не сохраняется;
- для каждого изображения генерируется уникальное имя, чтобы Jira не перепутала
  одноимённые вложения.

### Backend

Добавить endpoint:

```text
POST /api/jira/attachments
```

Backend должен:

- проверить домен Jira и ссылку на задачу;
- принять несколько файлов через `multipart/form-data`;
- передать их в API вложений Jira;
- установить необходимые заголовки Atlassian;
- вернуть идентификаторы, имена и URL вложений;
- ограничить размер и разрешённые MIME-типы;
- не сохранять файлы и токены на диске.

### Экспорт

- Data Center: формировать Wiki-ссылки на вложения, например `!file.png|thumbnail!`;
- Cloud: создавать ADF media nodes с корректным идентификатором вложения;
- перед созданием комментария проверять, что все используемые изображения загружены;
- при частичном сбое не публиковать неполный комментарий;
- предоставить повторную загрузку только неуспешных файлов.

### Дополнительный ручной режим

Если автоматическая загрузка недоступна:

- скачать все изображения одним ZIP-архивом;
- сохранить Jira-совместимые уникальные имена;
- сформировать текст со ссылками на эти имена;
- показать пользователю порядок действий для ручной пакетной загрузки.

## 6. Импорт существующего комментария по ссылке

### Пользовательский сценарий

Если отчёт о тестировании уже опубликован в Jira, пользователь:

1. копирует прямую ссылку на комментарий;
2. выбирает «Импорт из Jira по ссылке»;
3. вставляет ссылку;
4. приложение получает комментарий через Jira API;
5. преобразует его в редактируемые разделы, таблицы, статусы, код и изображения.

Пример ссылки может содержать:

- адрес Jira;
- ключ задачи;
- идентификатор комментария;
- якорь или параметры, по которым можно определить комментарий.

### Backend

Добавить endpoint:

```text
POST /api/jira/import-comment
```

Backend должен:

- проверить, что ссылка относится к Jira из настроек;
- извлечь ключ задачи и идентификатор комментария;
- получить комментарий через Jira REST API;
- для Data Center вернуть Wiki-разметку или доступное представление body;
- для Cloud вернуть ADF-документ;
- не передавать токен обратно во frontend;
- выдавать понятные ошибки при отсутствии доступа, удалённом комментарии или
  неподдерживаемом формате ссылки.

### Переработка импортера

Текущий импорт Wiki-разметки необходимо заменить на отдельный слой нормализации:

```text
Jira Wiki Markup ─┐
                  ├─> Normalized report document ─> Visual editor
Jira Cloud ADF ───┘
```

Новый импорт должен:

- считать каждую таблицу отдельным разделом;
- сохранять порядок блоков комментария;
- поддерживать таблицы с разным составом колонок;
- определять номер и статус независимо от положения столбцов;
- сохранять пустые ячейки;
- распознавать заголовки, абзацы, списки, ссылки и переносы строк;
- восстанавливать цвета и значения статусов;
- поддерживать кодовые блоки после появления структурированных ячеек;
- импортировать ссылки на Jira-вложения и изображения;
- корректно обрабатывать экранированные символы таблиц;
- не объединять соседние таблицы без явного основания;
- не терять неизвестные блоки молча.

### Отчёт об импорте

После разбора показывать пользователю результат:

- сколько разделов, строк и колонок найдено;
- какие элементы импортированы полностью;
- что было преобразовано в обычный текст;
- какие элементы не поддерживаются;
- какие изображения требуют загрузки или доступа;
- предупреждение перед заменой текущего черновика.

Желательно добавить режим предварительного сравнения:

- текущий документ;
- импортируемый документ;
- «Заменить текущий»;
- «Добавить как новые разделы».

### Совместимость

Проверить отдельно:

- Jira Cloud ADF;
- Jira Data Center Wiki Renderer;
- старые комментарии с нестандартной разметкой;
- комментарии, отредактированные визуальным редактором;
- ссылки на конкретный комментарий разных форматов;
- комментарии с несколькими таблицами;
- комментарии с вложениями, кодом и смешанным текстом.

## 7. Локальная история отчётов

### Пользовательский сценарий

В хедере появляется компактная кнопка «История». Она открывает локальную библиотеку
сохранённых отчётов.

Пользователь может:

- открыть старый отчёт;
- продолжить его редактирование;
- создать копию старого отчёта;
- переименовать запись;
- найти отчёт по ссылке или ключу задачи;
- удалить отдельный отчёт;
- очистить всю историю;
- увидеть дату создания и последнего изменения.

Перед открытием старого отчёта текущий черновик должен автоматически сохраняться, чтобы
переход не приводил к потере данных.

### Что хранить

Для каждого отчёта:

```text
StoredReport
 ├─ id
 ├─ title
 ├─ issueUrl
 ├─ issueKey
 ├─ environment
 ├─ overallStatus
 ├─ createdAt
 ├─ updatedAt
 ├─ lastOpenedAt
 ├─ document
 ├─ attachmentIds[]
 └─ schemaVersion
```

Название записи можно формировать автоматически:

```text
STUD-123 — STAGE — 19.06.2026
```

Пользователь должен иметь возможность изменить его вручную.

### Хранение

Для истории использовать IndexedDB, а не `localStorage`:

- она рассчитана на хранение нескольких документов;
- поддерживает поиск и индексы;
- позволяет в будущем хранить изображения;
- не блокирует страницу при работе с большими отчётами.

`localStorage` можно оставить только для:

- идентификатора текущего открытого отчёта;
- темы;
- небольших пользовательских настроек.

### Автосохранение и версии

Разделить два понятия:

- **отчёт** — самостоятельный документ в библиотеке;
- **версия** — сохранённое состояние этого отчёта во времени.

Для MVP истории:

- автоматически сохранять текущий отчёт после изменений;
- хранить только последнее состояние каждого отчёта;
- создавать отдельный снимок перед импортом, массовым удалением и публикацией в Jira.

В расширенной версии:

- хранить несколько версий одного отчёта;
- показывать временную шкалу;
- восстанавливать выбранную версию;
- создавать копию из старой версии.

### Ограничения хранения

В настройках истории предусмотреть:

- максимальное количество отчётов, например 50;
- максимальный срок хранения, например 90 дней;
- максимальное количество версий одного отчёта;
- отображение занятого объёма;
- автоматическое удаление самых старых записей;
- возможность закрепить важный отчёт, чтобы он не удалялся автоматически.

Значения по умолчанию:

- 50 отчётов;
- 90 дней;
- 10 снимков на отчёт;
- предупреждение при приближении к браузерной квоте.

### Очистка истории

Должны быть доступны:

- удаление одного отчёта;
- удаление всех незакреплённых отчётов;
- удаление отчётов старше выбранного срока;
- полная очистка истории;
- отдельная очистка неиспользуемых изображений.

Полная очистка является необратимым действием, поэтому требуется:

- явное подтверждение;
- указание количества удаляемых отчётов;
- предупреждение об удалении локальных изображений;
- предложение экспортировать резервную копию.

### Резервное копирование

Добавить:

- экспорт одного отчёта в JSON;
- экспорт всей истории в один архив;
- импорт резервной копии;
- проверку версии схемы перед восстановлением;
- защиту от дублирования идентификаторов.

Jira-токены и настройки авторизации никогда не должны попадать в резервную копию.

### Интерфейс истории

Окно истории должно содержать:

- поиск;
- сортировку по дате изменения, создания и ключу задачи;
- фильтр по окружению и итоговому статусу;
- карточки или компактную таблицу отчётов;
- действия «Открыть», «Создать копию», «Удалить»;
- кнопку «Очистить историю»;
- индикатор занятого объёма.

Кнопку «История» разместить в хедере рядом с действиями «Новый» и «Импорт».

### Безопасность и ограничения

- история хранится только в браузере конкретного устройства;
- очистка данных браузера удалит историю;
- разные браузеры и профили имеют независимую историю;
- для командной синхронизации в будущем потребуется серверное хранилище и авторизация;
- приватные данные отчётов не должны отправляться на сервер без отдельного решения.

## 8. Предлагаемый порядок реализации

### Этап 1. Быстрые улучшения интерфейса

1. Исправить тёмную тему предпросмотра.
2. Добавить полноэкранный режим.
3. Добавить историю undo/redo для структурных операций.
4. Расширить undo/redo на ввод текста.

### Этап 2. Структурированный контент ячеек

1. Изменить модель содержимого ячейки.
2. Добавить блоки кода.
3. Обновить импорт и оба экспортёра Jira.
4. Добавить миграцию существующих локальных черновиков.

### Этап 3. Изображения

1. Локальная вставка и IndexedDB.
2. Предпросмотр изображений в таблице.
3. Backend-загрузка вложений.
4. Экспорт Data Center.
5. Экспорт Cloud ADF.
6. Повторные попытки, ограничения и обработка частичных ошибок.

### Этап 4. Импорт опубликованных отчётов

1. Переработать импорт через нормализованную модель документа.
2. Добавить полноценный ADF-импорт.
3. Улучшить Wiki Markup-импорт.
4. Реализовать извлечение комментария по ссылке.
5. Добавить отчёт о потерях и предварительное сравнение.
6. Поддержать код и вложения после реализации соответствующих типов блоков.

### Этап 5. История отчётов

1. Ввести идентификатор отчёта и версию схемы документа.
2. Подключить IndexedDB.
3. Реализовать библиотеку отчётов и автосохранение.
4. Добавить открытие, копирование и удаление отчёта.
5. Добавить лимиты и автоматическую очистку.
6. Добавить экспорт и импорт резервной копии.
7. Связать изображения с жизненным циклом отчётов.

## 9. Критерии готовности

Функции считаются готовыми после проверки:

- undo/redo не повреждает таблицы и работает после сложной последовательности операций;
- полноэкранный режим корректно работает в светлой и тёмной темах;
- код сохраняет отступы после сохранения, импорта и публикации;
- изображения открываются в Jira в полном размере;
- комментарий не публикуется, если хотя бы одно обязательное вложение не загрузилось;
- Cloud и Data Center проверены отдельно;
- старые черновики корректно мигрируют в новую модель данных.
- комментарий можно импортировать по прямой ссылке без ручного копирования разметки;
- импорт не объединяет независимые таблицы;
- пользователь видит предупреждение обо всех неподдерживаемых элементах.
- старый отчёт можно открыть без потери текущего черновика;
- история выдерживает установленный лимит и корректно очищает старые записи;
- полная очистка требует подтверждения;
- экспортированная резервная копия восстанавливается без потери структуры;
- удаление отчёта не удаляет изображения, которые используются другими отчётами.

## Статус реализации

Реализовано:

- отмена и повтор действий, включая сочетания клавиш;
- тёмный предпросмотр таблиц;
- фокус-режим чек-листа;
- кодовые блоки внутри ячеек и экспорт в Wiki Markup/ADF;
- JSON-подсветка и горизонтальная прокрутка длинного кода;
- вставка изображений из файлов и буфера обмена;
- загрузка изображений во вложения Jira перед публикацией;
- импорт комментария по прямой ссылке;
- импорт Jira Cloud ADF и Jira Data Center Wiki Markup;
- отчёт о результате импорта;
- замена текущего отчёта или добавление импортированных разделов;
- локальная история отчётов в IndexedDB;
- восстановление, копирование, удаление и очистка истории.
- ручное изменение ширины колонок;
- закреплённая панель форматирования в обычном и фокус-режиме;
- добавление разделов в фокус-режиме.

Требует проверки пользователем на реальной Jira:

- форматы прямых ссылок на комментарии конкретной конфигурации Jira;
- доступ к изображениям и вложениям при импорте;
- загрузка нескольких вложений;
- отображение кодовых блоков и изображений в опубликованном комментарии;
- особенности Wiki Renderer в Jira Data Center;
- поддержка ADF media на конкретной версии Jira Cloud.
- визуальное представление attachment-card для загруженных файлов в Jira Cloud.
