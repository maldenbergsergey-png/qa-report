# QA Report

Автономный web-редактор QA-чек-листов с экспортом в Jira Wiki Markup.

## Запуск

Для функций Jira приложение нужно запускать через встроенный Node.js-сервер:

```bash
npm start
```

Затем откройте `http://localhost:4173`.

Если npm недоступен, эквивалентный запуск:

```bash
node server.js
```

Простое открытие `index.html` подходит только для автономного редактора. Проверка подключения
и отправка комментариев требуют `server.js`.

## Возможности MVP

- ввод метаданных отчёта;
- форматированный вводный текст;
- несколько разделов с таблицами;
- разделение существующей таблицы на новый раздел с выбранной строки;
- перестановка разделов кнопками и перетаскиванием;
- добавление, дублирование, перемещение и удаление проверок;
- произвольный состав колонок в каждом разделе: добавление, переименование и удаление;
- редактирование названий и действия с колонками непосредственно в заголовке таблицы;
- обязательными остаются только номер и статус;
- пять статусов из инструкции команды;
- цветовая индикация общего результата;
- автоматическая сводка;
- локальное автосохранение;
- импорт существующего Jira Wiki Markup, включая несколько таблиц с разными колонками;
- визуальный предпросмотр;
- экспорт и копирование Jira Wiki Markup.
- светлая и тёмная темы с запоминанием выбора.
- undo/redo с кнопками и горячими клавишами;
- фокус-режим для работы только с таблицами;
- локальная история отчётов в IndexedDB;
- кодовые блоки внутри ячеек;
- вставка изображений из файлов и буфера обмена;
- ручное изменение ширины каждой пользовательской колонки;
- закреплённая под хедером панель форматирования, доступная в фокус-режиме;
- импорт по прямой ссылке на комментарий Jira;
- загрузка изображений во вложения перед публикацией комментария.
- встроенная обратная связь со скриншотами из файлов и буфера обмена.

Пустые значения экспортируются как визуально пустые Jira-ячейки с сохранением структуры
таблицы.

## Интеграция с Jira

В интерфейсе доступны «Настройки Jira» и «Отправить в Jira».

- Jira Data Center / Server: адрес Jira и Personal Access Token. Комментарий отправляется
  через REST API v2 в формате Jira Wiki Markup.
- Jira Cloud: адрес Jira, email Atlassian и API token. Комментарий отправляется через REST
  API v3 в Atlassian Document Format.
- Адрес, тип Jira и пользователь сохраняются локально.
- На странице чек-листа указывается полная ссылка на задачу; ключ извлекается из неё
  автоматически при публикации.
- Токен не записывается в `localStorage` и существует только в памяти вкладки.
- Запросы выполняет локальный backend, поэтому токен не встраивается в исходный код и
  отсутствуют браузерные CORS-ограничения.

Для Jira Data Center / Server доступны два способа авторизации:

- Personal Access Token — для версий Jira, поддерживающих PAT;
- логин и пароль через HTTP Basic Auth — для старых установок, включая Jira 7.x.

Пароль и токен хранятся только в памяти открытой вкладки и исчезают после перезагрузки.
Пользователь должен иметь права просмотра задачи, добавления комментариев и вложений.

После отправки приложение обязательно проверяет созданный комментарий повторным запросом к
Jira. Успех показывается только при получении идентификатора комментария и возможности
прочитать его обратно. Если прокси вернул успешный статус без объекта комментария,
приложение пытается найти публикацию среди последних комментариев задачи.
При неудаче ответ содержит диагностические данные: upstream HTTP-статус, `Location`,
краткое тело ответа, количество комментариев до и после публикации и ошибки контрольного
чтения. В интерфейсе такая ошибка показывается дольше и полностью доступна во всплывающей
подсказке.

Frontend проверяет ревизию backend API. Если статические файлы обновились, а Node-процесс
продолжает работать со старым `server.js`, вместо ложной ошибки Jira будет показано
требование перезапустить процесс или пересобрать контейнер.

Если корпоративный WAF/SSO возвращает HTML browser challenge вместо JSON Jira, приложение
останавливает публикацию до POST и возвращает `JIRA_SECURITY_CHALLENGE`. Для серверной
интеграции администратор должен разрешить исходящий IP приложения либо исключить
`/rest/api/*` из интерактивной browser-проверки при наличии корректного заголовка
`Authorization`. Cookie браузерной Jira-сессии приложением не копируются и не сохраняются.

Проверка адаптеров подготовлена в `tests/jira-integration.test.js`; тест использует локальную
имитацию Jira и не обращается во внешнюю систему.

При публикации локальные изображения сначала загружаются во вложения задачи, после чего
создаётся комментарий со ссылками на загруженные файлы.

- Jira Data Center перед загрузкой читает имена уже существующих вложений задачи и выбирает
  следующий свободный вариант: `screenshot-1.png`, `screenshot-2.png` и далее. В комментарий
  изображение вставляется как Jira-вложение `!screenshot-N.png|thumbnail!`, поэтому миниатюра
  открывается во встроенном просмотрщике и не подменяет изображения из старых отчётов.
  Если список вложений недоступен по правам, к имени добавляется временная метка.
  При повторной публикации локальные изображения загружаются заново с новыми свободными
  именами: новый комментарий продолжит работать, даже если прежние вложения удалили из Jira.
  Код передаётся через универсальную обёртку `{code}...{code}` без указания языка:
  это совместимо в том числе со старыми Jira, где форматтер `json` отсутствует.
- Jira Cloud получает вложения как валидные ADF attachment-cards (`inlineCard`), а код как
  ADF `codeBlock`. REST attachment ID нельзя напрямую использовать как Media Services ID
  в `mediaGroup`: Jira отклоняет такой комментарий с `ATTACHMENT_VALIDATION_ERROR`.
  Поэтому в Cloud используется кликабельная карточка вложения; гарантированное встроенное
  thumbnail-превью требует Media Services UUID, которого стандартный attachment API не
  возвращает.
- Перед отправкой приложение проверяет сигнатуру изображения, приводит MIME-тип и
  расширение в соответствие и загружает файлы по одному. При ошибке отображается имя
  конкретного файла, его тип и размер.

После обновления Docker-развёртывания пересоберите контейнер:

```bash
docker compose up -d --build
```

Форматы ссылок на комментарии, права на вложения и итоговое отображение необходимо
проверить на используемой версии Jira Cloud или Data Center.

## Обратная связь

Кнопка с иконкой жука в хедере открывает форму обратной связи. Обращение всегда сохраняется
на сервере в каталоге `feedback-data`: описание и техническая информация находятся в
`feedback.json`, скриншоты сохраняются отдельными файлами. Текущий чек-лист прикладывается
только при явно включённой пользователем галочке; настройки Jira, пароль и токен в форму не
передаются.

В Docker каталог хранится в постоянном volume `feedback_data`, поэтому обращения не
исчезают при пересборке контейнера.

Для дополнительной отправки на email используется Resend. API key хранится как Docker
secret и не попадает в image, git или переменные окружения контейнера. Создайте файл
`secrets/resend_api_key`, выдайте доступ только владельцу и поместите в него ключ:

```bash
mkdir -p secrets
chmod 700 secrets
printf '%s' 're_...' > secrets/resend_api_key
chmod 600 secrets/resend_api_key
```

Адреса остаются в `.env`:

```env
FEEDBACK_TO_EMAIL=you@example.com
FEEDBACK_FROM_EMAIL=QA Report <feedback@your-domain.ru>
```

`FEEDBACK_FROM_EMAIL` требует подтверждённого в Resend домена. Для первой проверки его
можно не указывать: будет использован тестовый отправитель Resend. После изменения `.env`
пересоберите контейнер:

```bash
docker compose up -d --build
```

В подготовленной папке проекта `.env` и Docker secret уже подключаются автоматически.
Достаточно скопировать папку на сервер целиком и выполнить команду выше — вводить почтовые
настройки в интерфейсе не требуется. При запуске через `npm start` сервер также
автоматически читает `.env` и `secrets/resend_api_key`.

Запланированные улучшения и этапы развития описаны в `ROADMAP.md`.

## Развёртывание через Docker

На сервере скопируйте папку проекта и выполните:

```bash
docker compose up -d --build
```

Приложение будет доступно по адресу:

```text
http://SERVER_IP:4173
```

Чтобы изменить внешний порт:

```bash
cp .env.example .env
```

Измените `APP_PORT` в `.env`, затем перезапустите контейнер:

```bash
docker compose up -d --build
```

Проверка состояния:

```bash
docker compose ps
curl http://localhost:4173/api/health
```

Просмотр журналов:

```bash
docker compose logs -f qa-report
```

Остановка:

```bash
docker compose down
```

Для публичного сервера рекомендуется разместить контейнер за HTTPS reverse proxy
(Nginx, Caddy или Traefik). Это особенно важно, поскольку приложение передаёт Jira-токен
между браузером и собственным backend.

## Редактор кода и таблиц

- Столбцы перемещаются через меню или перетаскиванием за маркер; во время переноса
  подсвечиваются исходный столбец и сторона вставки.
- Для блоков кода автоматически определяется JSON, JavaScript, TypeScript, Python, SQL,
  Bash, HTML, CSS, YAML, Java и HTTP.
- Карточка кода показывает язык, количество строк и первые шесть строк с плавным
  затемнением. Короткий клик открывает полноэкранное редактирование, а код можно
  скопировать сразу с Jira-обёрткой `{code}...{code}`.
- Перед блоком и после него создаются обычные текстовые строки.
- Панели действий изображения и кода появляются только при наведении или фокусе.
  Короткий клик открывает увеличенный просмотр, drag-and-drop перемещает элемент и не
  открывает окно после отпускания.
- Изображение копируется как картинка через панель действий. Дополнительное меню позволяет
  скачать и выровнять изображение или растянуть его по ширине ячейки. Если браузер
  запрещает бинарное копирование, в буфер помещается Jira-разметка `!filename|thumbnail!`
  либо ссылка на файл.
- Изображения и блоки кода можно перетаскивать между ячейками. Изображение также можно
  выделить через меню и изменить по ширине за нижний правый угол.
- Кнопка копирования кода записывает два представления: Jira wiki-разметку в `text/plain`
  и внутренний HTML-сниппет. Поэтому вставка в Jira остаётся текстовой, а вставка обратно
  в редактор автоматически восстанавливает блок кода. Обычная разметка
  `{code}...{code}` и `{code:language}...{code}` при вставке также распознаются.
