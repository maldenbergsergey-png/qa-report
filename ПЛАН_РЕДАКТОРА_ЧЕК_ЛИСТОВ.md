# План разработки редактора QA-чек-листов для Jira

## 1. Что требуется сделать

Нужен отдельный полноэкранный редактор отчётов о тестировании, в котором QA может:

- писать обычный форматированный текст;
- создавать несколько логических блоков с заголовками;
- добавлять и редактировать таблицы;
- свободно добавлять, удалять, перемещать и дублировать строки и столбцы;
- менять набор столбцов под конкретную задачу;
- выбирать статус проверки из фиксированного списка;
- вставлять изображения в ячейки, прежде всего в «Фактический результат»;
- видеть итоговый отчёт примерно так, как он будет выглядеть в Jira;
- получить готовый текст для вставки в режим Jira «Текст»;
- в расширенном варианте — загрузить вложения и отправить готовый комментарий в Jira автоматически.

Инструмент должен учитывать правила из инструкции:

- обязательные поля: номер, проверка, фактический результат и статус;
- рекомендуемое поле: ожидаемый результат;
- необязательное поле: комментарий;
- разрешены дополнительные столбцы;
- проверки могут быть разделены на логические блоки;
- статусы: `OK`, `НЕ ОК`, `ПОЧТИ ОК`, `НЕ ПРОВЕРЕНО`, `ЧАСТИЧНО ПРОВЕРЕНО`;
- изображения в отчёте должны ссылаться на вложения Jira и открываться в полном размере.

## 2. Главный технический вывод

Таблицу и форматированный текст можно преобразовать в Jira Wiki Markup и скопировать в буфер обмена. Например:

```text
||Номер||Проверка||Ожидаемый результат||Фактический результат||Статус||Комментарий||
|1.|Отображение поля|Поле отображается|Соответствует ожидаемому|{color:#14892c}*OK*{color}| |
```

С изображениями есть принципиальное ограничение: текстовый буфер обмена не может сам создать вложения в Jira. Сначала файл должен быть загружен в конкретную задачу, после чего в разметку можно вставить ссылку на его Jira-имя, например:

```text
!qa-001-login-form.png|thumbnail!
```

Поэтому нужны два режима работы.

### Режим A — без интеграции с Jira

- редактор формирует готовую Wiki-разметку;
- пользователь копирует её в режим Jira «Текст»;
- редактор отдельно подготавливает изображения с уникальными именами;
- пользователь загружает их в задачу одним пакетом;
- имена уже совпадают со ссылками внутри сформированного отчёта.

Это самый простой и безопасный MVP. Полностью автоматической загрузки картинок в этом режиме не будет, но останутся только два понятных действия: загрузить пакет изображений и вставить текст.

### Режим B — с интеграцией с Jira

- пользователь указывает URL задачи или её ключ;
- приложение через Jira REST API загружает все изображения как вложения;
- получает подтверждённые имена вложений;
- обновляет ссылки в отчёте;
- создаёт комментарий через API либо копирует уже окончательную разметку.

Это даёт сценарий «Опубликовать в Jira», но требует согласования авторизации, CORS, версии Jira и политики безопасности компании.

## 3. Рекомендуемый формат продукта

Оптимальный первый вариант — внутреннее web-приложение, адаптированное под большой экран.

Рекомендуемый стек:

- **React + TypeScript + Vite** — интерфейс и сборка;
- **TipTap (ProseMirror)** — текстовый WYSIWYG-редактор и расширяемая модель документа;
- собственное расширение TipTap для QA-таблиц и статусов;
- **Zustand** — состояние интерфейса;
- **IndexedDB через Dexie** — локальные черновики и изображения;
- **React Hook Form + Zod** — настройки документа и проверка данных;
- **Vitest + Testing Library** — модульные тесты;
- **Playwright** — сценарии редактирования, копирования и восстановления черновиков;
- при интеграции с Jira: небольшой **Node.js/NestJS или Fastify BFF**.

Почему TipTap:

- поддерживает заголовки, абзацы, списки, ссылки, выделение, цвета, undo/redo;
- основан на структурированной модели документа, а не на ненадёжном HTML;
- таблицы можно расширить собственными командами;
- удобно написать отдельный экспортёр в Jira Wiki Markup;
- можно хранить один документ и получать из него визуальное представление, Wiki-разметку и JSON-черновик.

Не стоит делать основной редактор на `contenteditable` вручную: таблицы, выделение нескольких ячеек, история изменений и вставка контента быстро превратятся в отдельный большой проект.

## 4. Предлагаемый интерфейс

### Верхняя панель

- название отчёта;
- ключ задачи Jira;
- окружение: `DEV`, `STAGE`, `PROD` или произвольное;
- итог тестирования;
- кнопки «Сохранить», «Предпросмотр Jira», «Копировать для Jira»;
- позднее — «Опубликовать в Jira».

### Панель форматирования

Базовый набор, близкий к Jira:

- стили текста и заголовки;
- жирный, курсив, подчёркивание, зачёркивание;
- цвет текста;
- ссылка;
- маркированный и нумерованный список;
- цитата и моноширинный текст;
- вставка таблицы;
- вставка изображения;
- отмена и повтор действия.

Не обязательно визуально копировать Jira один в один. Важнее сохранить знакомый набор возможностей и одинаковый результат экспорта.

### QA-таблица

Стартовый шаблон:

1. Номер
2. Проверка
3. Ожидаемый результат
4. Фактический результат
5. Статус
6. Комментарий

Операции:

- добавить строку сверху или снизу;
- удалить и дублировать строку;
- перетащить строку;
- добавить, удалить и переименовать столбец;
- изменить ширину столбца;
- вставить несколько строк из Excel/Google Sheets;
- автоматическая нумерация;
- разбить отчёт на секции;
- свернуть секцию;
- применить один статус к выбранным строкам;
- сохранить таблицу как шаблон.

Статус лучше сделать не свободным текстом, а выпадающим списком с цветом:

- `OK` — зелёный;
- `НЕ ОК` — красный;
- `ПОЧТИ ОК` — голубой;
- `НЕ ПРОВЕРЕНО` — фиолетовый;
- `ЧАСТИЧНО ПРОВЕРЕНО` — отдельный согласованный цвет.

## 5. Работа с изображениями

При вставке или перетаскивании изображения редактор:

1. сохраняет оригинальный файл в черновике;
2. показывает превью в нужной ячейке;
3. генерирует уникальное безопасное имя, например  
   `QA-1234-check-07-actual-01.png`;
4. хранит связь «элемент документа → файл → Jira-имя»;
5. при экспорте создаёт Jira-ссылку на это имя.

Нужны:

- вставка из буфера обмена;
- drag-and-drop нескольких файлов;
- подпись;
- удаление и замена;
- просмотр оригинала;
- контроль размера и формата;
- предупреждение о совпадающих именах;
- пакетная выгрузка файлов для ручной загрузки в Jira.

Для MVP можно сделать кнопку «Скачать все вложения». Лучше отдавать ZIP и рядом показывать короткую инструкцию загрузки.

## 6. Модель документа

Черновик рекомендуется хранить не как HTML, а как JSON:

```text
Report
 ├─ metadata
 │   ├─ title
 │   ├─ jiraIssueKey
 │   ├─ environment
 │   └─ overallResult
 └─ blocks[]
     ├─ paragraph
     ├─ heading
     ├─ list
     └─ checklistSection
         ├─ title
         ├─ columns[]
         └─ rows[]
             ├─ cells[]
             ├─ status
             └─ attachments[]
```

Это позволит:

- менять внешний вид без миграции содержимого;
- валидировать обязательные поля;
- делать разные экспортёры;
- сохранять версии;
- поддержать Jira Data Center и Jira Cloud разными адаптерами.

## 7. Экспорт в Jira

Нужен отдельный модуль `JiraWikiSerializer`, который преобразует внутренний JSON в текст:

- заголовки;
- обычные абзацы;
- жирный и курсив;
- цвет;
- ссылки;
- списки;
- таблицы;
- переносы строк внутри ячеек;
- изображения-вложения;
- экранирование символов `|`, `{`, `}`, `[`, `]` и обратной косой черты.

Обязательно сделать два окна предпросмотра:

- визуальный отчёт;
- итоговая Jira-разметка.

Кнопка «Копировать для Jira» должна записывать в буфер только текстовую Wiki-разметку. Перед копированием запускается проверка:

- заполнены ли обязательные поля;
- нет ли изображений без подготовленного Jira-имени;
- нет ли неподдерживаемого форматирования;
- нет ли слишком больших файлов;
- все ли статусы допустимы.

Важно: показанный на скриншотах режим `Визуальный / Текст` и Wiki-разметка характерны для конкретной конфигурации Jira. Перед разработкой экспортёра нужно зафиксировать версию Jira и используемый renderer. Jira Cloud с новым редактором использует другую модель документа, поэтому для него потребуется отдельный экспортёр.

## 8. Интеграция с Jira

### Вариант 1 — backend/BFF, рекомендуется для командного решения

Браузер отправляет файлы и отчёт внутреннему серверу. Сервер:

- авторизуется в Jira;
- загружает вложения;
- создаёт комментарий;
- возвращает ссылки и результат операции.

Плюсы:

- токены не находятся в браузере;
- можно централизовать аудит и права;
- не зависит от браузерных CORS-ограничений;
- проще поддерживать несколько пользователей.

Минусы:

- нужен внутренний сервис и его эксплуатация;
- необходимо согласование с безопасностью.

### Вариант 2 — браузерное расширение

Расширение работает на странице вашей Jira и использует текущую пользовательскую сессию.

Плюсы:

- удобная интеграция с открытой задачей;
- можно автоматически определить ключ задачи;
- потенциально можно вставлять результат прямо в редактор комментария.

Минусы:

- сложнее сопровождать;
- зависит от DOM и обновлений Jira;
- нужны корпоративная установка и разрешения;
- такой вариант лучше делать после стабильного standalone-редактора.

### Вариант 3 — Jira-плагин

Отдельная панель или диалог прямо в Jira.

Плюсы:

- лучший пользовательский сценарий;
- доступ к контексту задачи и авторизации.

Минусы:

- самый дорогой вариант;
- реализация сильно зависит от Jira Cloud или Data Center;
- потребуются администраторы Jira и процесс установки.

Для начала рекомендуется standalone web-приложение, затем BFF-интеграция. Плагин имеет смысл только после подтверждения пользы продукта.

## 9. Этапы реализации

### Этап 0. Техническая разведка

- определить Jira Cloud или Data Center/Server;
- зафиксировать версию Jira;
- проверить точный синтаксис таблиц и изображений на тестовой задаче;
- выяснить доступные способы авторизации;
- проверить возможность REST-загрузки вложений и создания комментария;
- согласовать лимиты вложений и требования безопасности;
- собрать 5–10 реальных отчётов команды как тестовые примеры.

Результат: короткая спецификация совместимости с Jira.

### Этап 1. Прототип редактора

- полноэкранная страница;
- текстовые блоки и заголовки;
- одна или несколько QA-таблиц;
- операции со строками и столбцами;
- статусы и цвета;
- локальное сохранение;
- генерация Wiki-разметки;
- копирование в буфер;
- предпросмотр.

Результат: можно составить реальный чек-лист и вставить его в Jira без изображений.

### Этап 2. MVP с изображениями

- вставка, перетаскивание и хранение изображений;
- превью внутри ячейки;
- уникальные Jira-имена;
- ссылки на вложения в Wiki-разметке;
- пакетная выгрузка вложений;
- проверка отчёта перед копированием;
- восстановление черновика после перезапуска браузера.

Результат: удобный ручной сценарий «загрузить подготовленные вложения → вставить готовый текст».

### Этап 3. Командная версия

- backend и база данных;
- вход через корпоративную авторизацию;
- список отчётов;
- автосохранение и версии;
- совместный доступ;
- шаблоны команды;
- экспорт/импорт JSON;
- права просмотра и редактирования.

### Этап 4. Публикация в Jira

- загрузка вложений через REST API;
- создание или обновление комментария;
- обработка ошибок и повторная отправка;
- защита от дублирования вложений и комментариев;
- журнал публикаций;
- переход к опубликованному комментарию.

## 10. Тестирование

Нужно проверить:

- длинные таблицы и десятки секций;
- вставку текста с символами Jira-разметки;
- многострочный текст в ячейке;
- кириллицу и смешанный русский/английский текст;
- копирование из Excel и Google Sheets;
- вставку изображений из буфера;
- одинаковые имена файлов;
- удаление изображения после генерации разметки;
- восстановление черновика;
- отмену/повтор после операций с таблицей;
- корректность каждого статуса и цвета в Jira;
- открытие изображения в полном размере;
- большие отчёты, ограничения комментария и вложений;
- частичный сбой при загрузке нескольких файлов.

Полезно завести набор эталонных документов и автоматически сравнивать с ожидаемой Wiki-разметкой.

## 11. Риски

- Jira Cloud и Jira Data Center используют разные редакторы и форматы комментариев.
- Конкретный renderer может быть настроен администратором Jira.
- Некоторые Jira-макросы внутри ячеек требуют отдельного экранирования.
- Автоматическая загрузка файлов невозможна только средствами буфера обмена.
- Прямая работа браузера с Jira API может блокироваться CORS и корпоративной аутентификацией.
- Хранение Jira-токена во frontend недопустимо для командного решения.
- Если делать браузерное расширение слишком рано, оно будет зависеть от конкретной верстки Jira.

## 12. Рекомендуемое решение

Начать с локального web-MVP:

1. React + TypeScript + TipTap.
2. Структурированный JSON-документ.
3. Полноценный редактор QA-таблиц.
4. Экспорт в Jira Wiki Markup.
5. Локальные черновики в IndexedDB.
6. Изображения с заранее подготовленными уникальными именами.
7. Пакетная выгрузка вложений.

После проверки MVP на реальных задачах добавить внутренний BFF и кнопку «Опубликовать в Jira».

Такой порядок быстро решает основную боль — тесный редактор Jira — и не блокирует запуск сложностями авторизации и корпоративной инфраструктуры.

## 13. Что нужно уточнить перед началом разработки

1. Jira Cloud это или Jira Data Center/Server, и какая версия?
2. Доступен ли REST API из внутренней сети?
3. Можно ли получить отдельный технический аккаунт или OAuth-приложение?
4. Нужно ли только копирование или обязательна публикация комментария одной кнопкой?
5. Нужна ли совместная работа, или достаточно локальных черновиков каждого QA?
6. Где приложение можно разместить: внутренний сервер, Kubernetes, статический хостинг?
7. Какие форматы и максимальные размеры изображений разрешены?
8. Должен ли инструмент поддерживать импорт существующей Wiki-разметки из Jira?
